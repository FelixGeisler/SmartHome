= ADR 2: Capability-based device state

== Status

Accepted

== Context

A device hub serves heterogeneous devices: some are switchable, some only report readings
such as temperature or humidity, and others differ again. Modeling every device as a
switchable plug (a boolean `on` flag, a toggle-only command path, toggle-card UI) cannot
represent a device with no on/off notion. It turns every new device kind into a
structural change across the schema, the commands, and the UI.

== Decision

A `DeviceType` declares the set of capabilities it supports, such as `SWITCHABLE`. A
device's runtime state is a key/value snapshot in a `device_state` table rather than a
typed column per behavior. Capability gates commands: a toggle requires
`SWITCHABLE`, and a device without it gets a 422 problem response. The REST contract
exposes `capabilities` and `state`, and the dashboard chooses controls per capability,
rendering a toggle card for a switchable device and a read-only readings list otherwise.

== Consequences

Adding a device kind is additive: it declares its capabilities instead of inheriting a
fixed device shape, so a read-only sensor is one new enum constant and a card variant.
State values are untyped strings interpreted per capability (`on == "true"`), which the
database cannot validate, and the link between a device's state keys and its declared
capabilities is enforced only in code. Typing is per capability, so one capability can
adopt a richer representation without affecting the others.
