[frontend] --> [device] : REST (/api)

[frontend] --> [assistant] : chat (/api)
[assistant] ..> [device] : tool-use

[registry] --> port

[adapter] --> dev

[cols="1,3",options="header"]
|===
| Block | Responsibility
| `frontend` | The dashboard: a React SPA (own toolchain under `hub/frontend/`) that registers, lists, and controls devices, charts sensor history, and hosts the AI chat — all over the REST API. It only sees the REST contract, never the domain model.
| `web` | Cross-cutting web concerns: maps exceptions to RFC 9457 `ProblemDetail`, serves the SPA, and exposes the OpenAPI document.
| `device` | Application core: the device model plus register / list / control logic and persistence. It records inbound sensor readings and emits a reading event for the streaming side.
| `capability` | The neutral state and command vocabulary (on/off, brightness, color as CIE xy / Kelvin / %) that every adapter speaks, so the core never learns a vendor's model.
| `integration` | The `DeviceAdapter` port and the registry that routes to it; one sub-package per ecosystem (bridge/cloud and MQTT adapters) plugs in here.
| `settings` | Persisted, auto-restored integration connections and keys, so a paired bridge, broker, or assistant key survives a restart.
| `telemetry` | The sensor-history read side (served over the API from a search index) and the outbound publisher that streams reading events onto the pipeline.
| `assistant` | The AI control surface: it turns natural-language prompts into tool-use calls against the same device and telemetry services the UI uses.
|===
